# Bilibili Video Downloader Requirements (V1)

## 1. Product Goal
Build a command-line client first (MVP), then extend to desktop GUI in a later phase.

The MVP should provide an end-to-end flow:
1. Search videos by keyword.
2. Get basic metadata from a video URL.
3. Download video from a video URL.
4. Download audio from a video URL.

## 2. Scope

### 2.1 In Scope for V1 (CLI)
- `search`: search by keyword and display result list.
- `info`: parse URL and return basic metadata.
- `download-video`: download video with selectable quality/container.
- `download-audio`: download audio with selectable format/quality.

### 2.2 Out of Scope for V1
- Desktop GUI.
- Full comment crawling and advanced content analysis.
- Account login flows, paid-content unlocking, or DRM bypass.

### 2.3 V2 Direction (GUI)
- Add desktop interface while reusing the same core command/service layer.
- Keep feature parity with V1 commands.

## 3. Functional Requirements

### FR-1 Search by Keyword
**Input**
- User enters keyword or sentence.

**Process**
- Call Bilibili search API.
- Return a paginated list.

**Output fields**
- `title`
- `author`
- `duration`
- `play`
- `danmaku`
- `url`

**User decision point**
- User may continue to run `info`, `download-video`, or `download-audio` using the selected URL.

### FR-2 Basic Metadata by URL
**Input**
- User enters a Bilibili video URL.

**Process**
- Resolve video id (`bvid`/`aid`) from URL.
- Query video detail endpoint.

**Output fields (basic only)**
- `title`
- `description`
- `play_count`
- `danmaku_count`
- `url`

### FR-3 Video Download
**Input**
- Video URL
- Optional quality (`best`, `1080`, `720`, `480`, `360`)
- Optional container (`mp4`, `mkv`, `webm`)
- Optional output path

**Process**
- Fetch downloadable streams.
- Download and merge streams if needed.

**Output**
- Local video file saved under configured output directory.

### FR-4 Audio Download
**Input**
- Video URL
- Optional audio format (`mp3`, `m4a`, `aac`, `wav`, `flac`)
- Optional audio quality (kbps)
- Optional output path

**Process**
- Extract best available audio stream.
- Convert to target format if required.

**Output**
- Local audio file saved under configured output directory.

## 4. CLI Interaction Requirements
- Use subcommands: `search`, `info`, `download-video`, `download-audio`.
- Provide `--help` for all commands.
- Return clear error messages and non-zero exit code on failures.

## 5. Non-functional Requirements
- Retry and timeout strategy for network requests.
- Clear logs for progress and failures.
- Stable behavior for invalid URL, network timeout, and unavailable resources.
- Keep architecture ready for GUI reuse (separate core logic from UI later).

## 6. Error Handling Requirements
- Invalid URL: prompt user to provide a valid Bilibili video URL.
- API failure: show API error message.
- Download failure: show reason and keep partial-file cleanup policy documented.
- Missing dependency/tool: provide install hints.

## 7. Compliance and Constraints
- Follow applicable laws, platform terms, and copyright requirements.
- Only use the tool for content the user is legally authorized to access/download.

## 8. Acceptance Criteria (V1)
1. User can search with keyword and receive at least one page of structured results.
2. User can input a valid video URL and receive basic metadata.
3. User can download video with specified quality/container.
4. User can download audio with specified format/quality.
5. Invalid inputs and network errors are handled with readable messages and non-zero exit code.
6. CLI help is available for root command and all subcommands.
